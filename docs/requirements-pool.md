## 需求详情

### REQ-014 智能体模板原文权限说明优化

- 优先级：P1
- 状态：Todo
- 模块：Report / Privacy / Agent
- 背景：当前智能体模板使用 `requiresRawNotesConsent` 表达是否必须授权读取划线原文和个人想法。用户确认当前阶段不做 required / optional / none 三档拆分，继续保持两类策略：必需授权和可选授权。
- 产品结论：
  - 隐私确认必须继续严格：只要读取划线原文、书摘原文或个人想法，就必须由用户显式确认。
  - `requiresRawNotesConsent=true`：没有原文授权不能生成，适用于确实依赖原文证据的模板。
  - `requiresRawNotesConsent=false`：未授权也能生成，只使用统计、书架、笔记数量和分类等低敏数据；用户主动勾选后才读取划线原文和个人想法。
  - 当前阶段不新增 `rawNotesPolicy`，也不区分 optional / none；等模板数量增长或出现纯统计模板后再考虑拆分。
- 技术边界：
  - 继续使用 `requiresRawNotesConsent` 和 `rawNotesConsent` 两个布尔值，不做模型字段升级。
  - 必需授权模板未授权时继续后端硬校验。
  - 非必需授权模板未授权时不写入 `data/notes.raw.json`。
  - 非必需授权模板已授权时才预取原文划线和个人想法。
  - 用户自定义要求不能覆盖原文授权策略，不能通过 prompt 要求读取未授权数据。
- UI / 文案要求：
  - 不向用户展示 `requiresRawNotesConsent`、`rawNotesConsent` 等工程字段。
  - 必需授权模板使用“需要允许读取划线原文和个人想法后才能生成”。
  - 非必需授权模板使用“可不授权生成；允许后报告会更具体”。
  - 生成配置摘要要明确当前是否会使用划线原文和个人想法。
- 验收：
  - 必需授权模板未授权时无法开始生成，并给出用户可理解的原因。
  - 非必需授权模板未授权时可以生成，且 job 数据目录不包含 `notes.raw.json`。
  - 非必需授权模板授权后才预取原文划线和个人想法。
  - `input/user-policy.json` 或 `input/brief.md` 明确告知 Agent 当前是否可用原文；未授权时要求不要编造划线原文、个人想法、读后感或书籍原文。
  - 前端模板卡片和工作台用用户语言解释数据读取范围。
  - `npm run frontend:typecheck`、`npm run frontend:build`、`cd src-tauri && cargo check` 通过。

### REQ-015 划线样式与颜色信息利用

- 优先级：P1
- 状态：Todo
- 模块：Notes / Export / Report
- 背景：微信读书划线接口返回 `colorStyle`，当前项目已经在后端和前端类型中保留该字段，但未在 UI、Markdown 或报告中利用。Skill 文档只说明这是“划线颜色样式”，未给出稳定枚举，因此第一阶段不能把数值直接翻译成黄色、蓝色或波浪线等具体样式。
- 产品价值：
  - 帮助用户在导出后保留原本的标注层次。
  - 让 Notes 页更像“私人阅读账本”，但不引入编辑能力。
  - 为后续报告提供“标注习惯”信号，例如常用样式、不同样式的章节分布。
- 第一阶段范围：
  - Notes 页在划线卡片上展示低调的“标注样式”标记，可用色块、细线或标签表达，但文案只写“样式 1 / 样式 2”，不猜测颜色含义。
  - Markdown 导出可选保留样式信息，例如在划线后追加 `标注样式：1`；默认文案必须用户可理解，不暴露工程字段名。
  - 报告数据模型可统计不同样式数量，但基础报告是否展示需看真实样本是否有区分度。
  - 增加真实数据抽样记录，确认常见 `colorStyle` 值域和是否与微信读书 App 的视觉样式稳定对应。
- 暂不做：
  - 不做划线样式编辑。
  - 不在未校准前把样式映射为具体颜色名或波浪线。
  - 不为每种样式新增一套强视觉 token，先沿用 Quiet Reading Ledger 的克制标注表达。
- 验收：
  - 不破坏现有划线 / 想法浏览和 Markdown 导出。
  - 用户能在 UI 或导出中区分不同标注样式。
  - 文案不出现 `colorStyle` 这类工程字段。
  - 若真实样本不足以校准颜色语义，文档明确保留为“样式值”。
  - `npm run frontend:typecheck`、`npm run frontend:build`、`cd src-tauri && cargo check` 通过。

### REQ-016 Post-MVP 用户反馈驱动优化池

- 优先级：P2
- 状态：Todo
- 模块：Product / Release
- 背景：MVP 已完成，后续不按“完整微信读书客户端”补功能，而根据真实用户在导出、同步、报告和资料库整理中的反馈排序。
- 产品方向：
  - 导出可信度：重导出 / 重同步的差异说明、跳过原因、失败恢复和结果复核。
  - 内容可读性：Markdown 标题、章节结构、想法与划线的相对位置、Frontmatter 字段命名、ima 导入后的展示文案。
  - 报告复用：基础报告真实数据视觉回归、智能体报告隐私授权、输出形态质量提示和历史管理。
  - 个人资料库：Obsidian / ima / 本地文件夹的索引友好字段和使用说明。
- 暂不进入当前阶段：
  - PDF 导出。
  - 在线分享平台。
  - 完整书籍详情页、公共书评浏览、推荐发现、相似书推荐。
  - 在线阅读体验或微信读书客户端替代能力。
- 验收：
  - 发布后每个新增需求必须能对应到真实用户反馈、复现路径或明确场景。
  - P2 增强不能阻塞 Markdown 导出主链路。
  - 涉及隐私或外部服务的增强必须保留显式用户确认和失败隔离。

### REQ-007R 阅读报告真实数据回归与产物质量观察

- 优先级：P2
- 状态：Watch
- 模块：Report / QA
- 背景：阅读报告、智能体模板、输出形态和历史任务管理已完成可用闭环。后续不再维护长篇“已做了什么”清单，只按真实数据和用户反馈记录问题。
- 观察项：
  - 基础报告：长书名、排行密度、小窗口布局、历史周期数据口径。
  - 智能体报告：隐私授权说明、生成过程、历史记录、质量提醒、失败但有 HTML 时的可打开状态。
  - 输出形态：默认报告、PPT 风格、小红书图文风格的结构完整度和可读性。
- 新问题处理：
  - 可直接修的小问题作为当前需求附属修复。
  - 影响用户理解或数据可信度的问题，拆成新的 P1 / P2 需求。
